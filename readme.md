#### 什么时候逻辑在当前组件内写，什么时候在父组件写，逻辑组件和展示组件以什么标准分开等等
- 单一职责原则 一个组件应该只做一件事，并且把它做好
- 组合优于继承 通过组件嵌套和 props 传递来构建复杂 UI
- 状态提升 当多个组件需要共享状态时，将状态提升到它们的共同父组件
- 逻辑复用 将可复用的逻辑抽离成自定义 Hook
- 展示与容器分离 尽可能让展示组件保持纯净，专注于 UI，而容器组件专注于数据和逻辑
- 决策树
当你不确定一段逻辑该放在哪里时，可以问自己以下几个问题：
这个逻辑或状态是否被其他组件需要？
是 → 放在父组件或全局状态。这是状态提升的黄金法则。如果多个子组件需要访问同一数据或触发同一行为，那么这些数据和行为就应该属于它们的共同祖先。
否 → 放在当前组件。如果一个状态或逻辑只影响当前组件的 UI（例如，一个按钮的禁用状态、一个输入框的内容、一个下拉菜单的展开 / 折叠），那么它就应该是组件的私有状态。
这个逻辑是业务逻辑还是 UI 逻辑？
业务逻辑 (如：用户登录、获取商品列表、提交订单) → 放在容器组件或自定义 Hook 中。这些逻辑通常与数据和应用流程相关。
UI 逻辑 (如：表单验证反馈、动画触发、模态框显示 / 隐藏) → 放在展示组件中。这些逻辑通常只与组件自身的呈现有关。
这个逻辑是否需要被复用？
是 → 抽离成自定义 Hook。这是复用逻辑的最佳方式，无论它是业务逻辑还是 UI 逻辑。例如，我们的 useLocalStorage Hook。
否 → 根据前两个问题决定位置。
